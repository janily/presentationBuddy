## 二、修复方案

### 2.1 问题一：进度状态去重（左侧只保留一处，右侧只做摘要）

**设计原则**（沿用 04 号文档已确立、但未落实到位的原则）：**同一时刻，生成进度只允许存在一个"信息源"，其余地方只能引用它，不能各自渲染一份状态文案 + 进度条。**

**具体改动**：

1. `presentation-preview-pane.tsx`：
   - **删除** `generating` 分支里 `OutlinePreview` 的渲染，`generating` 阶段不再展示大纲卡片网格背景。
   - 保留 `GenerationProgress`，作为 `generating` 阶段左侧预览区的**唯一**内容（可以从"绝对定位悬浮在卡片网格上"改为在空白背景居中的独立卡片，参考 `presentation-processing-view.tsx` 现成的居中卡片样式，直接复用或抽取其 UI）。
   - 效果：`generating` 阶段左侧不再是"网格 + 悬浮进度条"的叠加视觉，而是一张干净的、居中的进度卡片。

2. `agent-panel.tsx` 的 `ProgressCard`：
   - 保留，但职责收窄为"摘要一句话 + 细进度条（当前实现已经是 4px 高度，符合要求）"，**不需要修改**，因为它和左侧的关系是"总-分"，不是重复，只要左侧不再重复展示相同的大文案即可消除"看起来像两个一样的进度条"的观感。

3. 验收标准：生成阶段截图应只能看到——左侧一张完整进度卡片（大标题+说明+进度条），右侧对话流一张摘要进度卡片（一行文案+细进度条），二者视觉样式/大小/层级要能明显区分主次，不能长得一样。

### 2.2 问题二：大纲抽屉必须"关得掉"

**改动文件**：`presentation-studio.tsx`（抽屉容器部分）

1. **给外层 header 加 `shrink-0`**，确保关闭按钮所在的行永远不会被内容挤压或滚动出视口：
   ```tsx
   <div className="flex shrink-0 items-center justify-between border-b border-[var(--border-light)] px-5 py-4">
   ```

2. **补齐三种关闭方式**（当前只有点击图标一种）：
   - 点击遮罩层（`bg-black/25` 那个 div）本身也要能关闭：给最外层遮罩 div 加 `onClick={() => setIsOutlineDrawerOpen(false)}`，内部抽屉面板要 `onClick={(e) => e.stopPropagation()}` 阻止冒泡。
   - 监听 `Escape` 键：抽屉打开期间监听 `keydown`，`Escape` 时关闭（`useEffect` 里 `document.addEventListener`，卸载/关闭时移除）。
   - 保留现有的 `X` 图标按钮，但把它的可点击区域从 `p-2` 增大到至少 `40x40px`（当前 `h-5 w-5` 图标 + `p-2` 大约是 36px，接近但建议明确写 `h-10 w-10` 的按钮容器，提升可点性）。

3. **视觉上区分"抽屉外层 header"和"OutlinePanel 内部 header"**，避免用户以为二者是一个东西：外层 header 标题可以保留"Edit outline"，但建议加一层浅色背景色块或把 `OutlinePanel` 内部 header 的字号/间距调小，形成"标题栏 → 内容区（内容区自带说明文字）"的清晰层级，而不是两个视觉权重相近的标题堆叠。

4. **回归测试点**：`reviewing`、`previewing` 两个阶段分别点击"Edit outline"/"View outline"打开抽屉，验证图标点击、遮罩点击、`Esc` 三种方式均可关闭，且关闭后 `isOutlineDrawerOpen` 状态正确复位（不会残留遮罩层）。

### 2.3 问题三：真正让 frontend-slides 发挥作用（核心修复，已确认技术路线）

> **结论：放弃"裸 Messages API + 文本提示假装调用 Skill"的方案，改用 Claude Agent SDK 在服务端以 headless agent 方式真正加载并执行 frontend-slides Skill。**

#### 2.3.1 为什么必须换路线

Skill 的本质是"一份放在 `.claude/skills/<name>/SKILL.md`（及配套资源文件）里的额外指令说明，在 Agent 运行时被发现并注入到系统提示/工具调用链路中"。这套发现与注入逻辑**只存在于 Claude Code CLI 和 Claude Agent SDK 的运行时里**，不存在于 Anthropic 官方或任何中转商的 `/v1/messages` 接口协议中。现有 `frontend-slides-invoker.ts` 绕开了运行时直接打 API，无论 prompt 里怎么写"请使用 xxx skill"，服务端收到的都只是一段普通文本，**该 Skill 的设计规范、模板、示例、CSS/动画库统统没有被真正加载**，模型只能凭参数记忆里"知道 frontend-slides 大概是什么"来发挥，效果和稳定性自然没有保障。

#### 2.3.2 目标架构

```
presentationHtmlGenerationStep (Mastra workflow step)
  → 调用新的 src/utils/frontend-slides-agent-runner.ts
      → 使用 @anthropic-ai/claude-agent-sdk 的 query()
      → 运行目录内置 .claude/skills/frontend-slides/SKILL.md（及其模板/资源）
      → 以 headless 模式（无需人工确认）运行一次性任务：
          "根据以下大纲生成 HTML 演示文稿" + 结构化大纲 JSON
      → 收集 agent 最终产出的完整 HTML 文本
  → 若 Agent SDK 调用失败/超时/未安装 → 降级到现有 presentationHtmlGenerationAgent（保留原逻辑作为兜底，不要删除）
```

#### 2.3.3 详细实施步骤

**步骤 1：引入依赖与安装 Skill 文件**

```bash
pnpm add @anthropic-ai/claude-agent-sdk
```

- 在项目根目录建立 `.claude/skills/frontend-slides/`，将 frontend-slides 仓库（`https://github.com/zarazhangrui/frontend-slides`）中的 `SKILL.md` 及其引用的模板、样式、脚本资源文件原样放入。
- **实习生行动项**：先手动跑一遍 `claude` CLI（本地已登录的账号）在一个测试目录里加载该 Skill、验证它确实能独立生成一份合格的 HTML 演示文稿，再进入下一步代码集成。避免在集成阶段才发现 Skill 本身有问题。

**步骤 2：新建 Agent SDK 调用封装**

新建文件：`src/utils/frontend-slides-agent-runner.ts`

```typescript
import { query } from "@anthropic-ai/claude-agent-sdk";
import type { FrontendSlidesInput } from "./outline-to-slides-mapper";

export type FrontendSlidesResult = { html: string };

export function isFrontendSlidesAgentConfigured() {
  // Agent SDK 走的是登录态或 ANTHROPIC_API_KEY，二选一即可，具体以 SDK 文档为准
  return Boolean(process.env.ANTHROPIC_API_KEY?.trim());
}

function buildTaskPrompt(input: FrontendSlidesInput) {
  // 复用现有 buildFrontendSlidesPrompt 的大纲拼装逻辑，
  // 但不再需要写"请使用 xxx skill"这种无效话术——
  // Skill 是否生效由 Agent SDK 运行时的技能发现机制决定，不依赖 prompt 措辞。
  return `根据以下批准的大纲生成一份完整的独立 HTML 演示文稿：

标题：${input.title}
风格：${input.style}
页数：${input.slides.length}

${input.slides.map((s, i) => `第 ${i + 1} 页：${s.title}\n${s.content}`).join("\n\n")}

要求：生成的 HTML 必须包含与大纲页数完全一致的幻灯片数量，不允许中途截断或省略任何一页。`;
}

export async function invokeFrontendSlidesAgent(
  input: FrontendSlidesInput,
): Promise<FrontendSlidesResult> {
  const prompt = buildTaskPrompt(input);
  let html = "";

  for await (const message of query({
    prompt,
    options: {
      // 关键：允许运行时发现并加载项目内的 .claude/skills
      cwd: process.cwd(),
      allowedTools: ["Skill"],
      permissionMode: "bypassPermissions", // headless 无人值守，禁止其触发写文件等副作用工具
      maxTurns: 6,
    },
  })) {
    if (message.type === "result" && message.subtype === "success") {
      html = extractHtmlFromAgentResult(message.result);
    }
  }

  if (!html) {
    throw new Error("frontend-slides agent did not return HTML content");
  }

  return { html };
}

function extractHtmlFromAgentResult(result: string): string {
  const fenced = result.match(/```(?:html)?\s*([\s\S]*?)```/i);
  if (fenced?.[1]) return fenced[1].trim();

  const tag = result.match(/(?:<!doctype html>|<html[\s>])[\s\S]*<\/html>/i);
  if (tag?.[0]) return tag[0].trim();

  throw new Error("Failed to extract HTML from frontend-slides agent result");
}
```

> 上面的 `query()` 参数（`allowedTools` / `permissionMode` / 消息结构）请以实际安装的 `@anthropic-ai/claude-agent-sdk` 版本文档为准做核对，字段名可能随版本变化，这里给出的是设计意图，不是可以直接照抄不检查的最终代码。

**步骤 3：workflow 接入 + 兜底策略不变**

- 修改 `src/mastra/workflows/presentation-generation-workflow.ts` 中 `presentationHtmlGenerationStep`：
  - 把 `invokeFrontendSlidesSkill` 换成 `invokeFrontendSlidesAgent`，`isFrontendSlidesConfigured` 换成 `isFrontendSlidesAgentConfigured`。
  - **保留**现有"失败则降级到 `presentationHtmlGenerationAgent`"的 try/catch 逻辑不动，作为双重保险。
  - 旧的 `frontend-slides-invoker.ts` 可以保留一段时间标记为 `@deprecated`，待新方案验收通过后再删除，避免出问题时无法快速回滚。

**步骤 4：解决"只生成一页"的兜底约束**

不管走 Agent SDK 还是任何方案，都必须补一条**硬性校验**，防止未来再次出现单页截断且没人发现：

```typescript
const slideSectionCount = (html.match(/<section[\s>]/gi) ?? []).length; // 或按 Skill 实际使用的分页标签调整
if (slideSectionCount < input.slides.length) {
  throw new Error(
    `frontend-slides output only contains ${slideSectionCount} slide(s), expected ${input.slides.length}`
  );
}
```

- 校验失败视为本次生成失败，走现有降级/重试逻辑，**不允许把明显缺页的 HTML 当作"生成完成"直接推给用户**。

#### 2.3.4 风险与应对

| 风险 | 应对 |
|-----|------|
| Claude Agent SDK 在部署环境（服务器/容器）里能否正常运行，是否需要额外的认证/网络出口配置 | 集成前先在预发布环境跑通一次最小可行请求，确认延迟和成功率，再合并到主流程 |
| Agent SDK 单次调用延迟明显高于裸 API（要跑完整 agent loop） | 复用现有 `FRONTEND_SLIDES_TIMEOUT_MS` 超时保护机制，必要时适当调大超时并在 UI 进度文案里注明"专业模式生成中，可能需要更长时间" |
| Skill 文件本身质量决定上限 | 集成前先用 CLI 独立验证 Skill 效果（见步骤 1），不要等集成完才发现 Skill 本身不行 |

## 三、验收标准

### 3.1 问题一
- [ ] `generating` 阶段左侧预览区只显示一张进度卡片，不再有大纲卡片网格与悬浮进度条并存的情况
- [ ] 右侧对话流的摘要进度卡与左侧进度卡视觉权重明显不同（一详一简），不会被误认为"重复的两个进度条"

### 3.2 问题二
- [ ] 点击图标 `X`、点击遮罩层空白处、按 `Esc` 键，三种方式均可关闭大纲抽屉
- [ ] 关闭按钮所在的 header 行任何情况下都不会被内容挤出可视区域（`shrink-0` 生效）
- [ ] 抽屉外层标题与 `OutlinePanel` 内部标题在视觉上可区分主次

### 3.3 问题三
- [ ] 生成的 HTML 幻灯片页数与批准大纲的页数完全一致，缺页直接判定失败并走降级/重试，不允许把缺页结果当成功展示
- [ ] 对比同一份大纲，新方案生成效果（视觉设计、动画、布局）应明显优于旧的裸 API 方案，需附生成效果截图对比
- [ ] Agent SDK 不可用/超时/异常时，能正确降级到现有 `presentationHtmlGenerationAgent`，用户侧看到的仍是一次完整生成，不会白屏或报错中断
