## 问题一：生成演示文稿失败（slide 数量校验报错）

### 现象

```
frontend-slides Mastra generation failed; no alternate generator is allowed:
frontend-slides output contains 20 slide(s), expected exactly 10
  at assertFrontendSlidesComplete (src/services/frontend-slides/html-validator.ts:35)
  at validateFrontendSlidesHtml (src/mastra/workflows/presentation-generation-workflow.ts:718)
```

用户要求 10 页，模型产出被判定为 20 页，初次与 repair 重试双双失败后整步报错（系统无备用生成器，见 `llmdoc/must/project-basics.md` 的"无备用生成器"不变量）。

### 根因分析

核心在 `countGeneratedSlides`（`src/services/frontend-slides/html-validator.ts:24-29`）：

```ts
const slideClassCount = html.match(/class=["'][^"']*\bslide\b[^"']*["']/gi)?.length ?? 0;
return slideClassCount || sectionCount;
```

1. **`\bslide\b` 的词边界会被连字符"击穿"**。正则 `\b` 把 `-` 视为非单词字符，所以 `class="slide-inner"`、`class="slide-content"`、`class="slide-number"` 等常见命名全部会被计为一张幻灯片。报错值恰好是期望值的 2 倍（20 vs 10），高度符合"每张 slide 外层 `class="slide"` + 内层一个 `slide-xxx` 容器"的典型 HTML 结构——**很可能实际只有 10 张幻灯片，是计数器误判**。
2. **prompt 没有约束 class 命名**。`src/services/frontend-slides/prompt-builder.ts` 只要求"exactly N 个 .slide 元素"，没有告知模型"除幻灯片根元素外，任何 class 都不得包含独立的 `slide` 词干"。模型按常规前端习惯写 `slide-title`/`slide-body` 完全合理。
3. **repair 重试大概率同样失败**。repair prompt 只是把失败原因（"contains 20 slide(s)"）回传，模型无从知道是"命名撞了计数正则"，重生成时仍会使用同类命名，于是两次全灭 → 整步抛错。
4. **校验是硬性 `!==` 相等**，任何计数误差直接判死，且失败成本极高（重新生成整个 60-100KB 文档）。

### 解决方案建议

按优先级排列：

1. **【必做】修正计数正则，只匹配独立的 `slide` class token**：把类属性拆成空白分隔的 token 列表后做全等比较（`token === "slide"`），或改用 `(?:^|\s)slide(?:\s|$)` 语义的匹配，彻底排除 `slide-content` 类误伤。这是纯逻辑 bug 修复，风险最低、收益最大。
2. **【建议】在契约 prompt 中显式声明计数规则**：告诉模型"幻灯片根元素必须且只能使用 `class="slide"`（可附加修饰 class，但整份文档中 class token 恰好为 `slide` 的元素数量必须等于 N）"，让生成端与校验端对齐同一份可判定契约。
3. **【建议】repair prompt 携带诊断信息而非仅错误消息**：失败时附上"当前匹配到的 class 列表/示例片段"，让第二次尝试有的放矢。
4. **【可选】校验失败时先尝试无损纠偏再重生成**：当 token 级精确计数等于 N 而旧正则计数不等时，说明是校验器误报，不应触发昂贵的整体重试。落地方式即方案 1，此条作为回归测试用例补充（现有 `html-validator.test.ts` 需要加 `slide-content` 混排的用例）。

> 注意：`assertFrontendSlidesDocument` 中第 57 行"missing .slide elements"检查用了同一个宽松正则，方案 1 修改时需同步评估，但那里只判"存在性"，宽松匹配无害。

## 问题二：生成过程中沟通调整内容，Agent 答非所问直接触发生成

### 现象（对照截图）

时间线：
1. 用户确认生成（10 页 Mastra 入门教程，Paper & Ink 风格），主生成流程启动，进度到 82%。
2. **生成尚未完成时**，用户在聊天框发送："需要加一页 mastra 框架和其它开发框架的对比"。
3. Agent 的文字回复本身是正确的修订提案（"插入一页对比、总页数变为 11、请回复'确认'以执行"），**但紧接着 UI 出现"正在生成演示文稿…"任务卡**——即用户还没确认，生成动作已被触发，整体表现为答非所问。

### 根因分析

这是"**生成进行中**"这个中间态在整条 chat → decision → action 链路上缺乏一致约束造成的，具体有三层叠加：

**(a) 修改请求发生时 deck 还不存在，提案链路整体失效。**
`/api/agent-chat` 只有在 `deckContext`（deckId + version）存在时才会 `createActionProposal` 并持久化（`route.ts:580-588`）；而 deck 要等主 workflow 完成、artifact 落库后才有。生成中的这条修改请求：
- `hasGeneratedDeck` 为 false → 服务端不注入"已有 deck，按修订讨论处理"的 system 提示（`route.ts:487-501`）；
- 没有 `deckContext` → 即便模型给出 `revise-structure` + revision，提案也不会被保存，后续用户回复"确认"时既没有 pending proposal（agent 被指示"没有 pending proposal 时绝不使用 execute-proposal"），走不了提案执行分支。

**(b) 模型在"新建 deck 决策策略"下工作，容易把上下文中已确认过一次的 brief 再次判为可生成。**
由于 (a)，模型套用的是"新 deck 决策策略"：对话历史里用户刚刚说过"确认生成"，最新一条又是具体明确的内容要求。instructions 中"确认后 readyToGenerate=true, nextAction=generate"的规则与"新增条件的确认不算干净确认"的规则在这个上下文里高度冲突，模型输出了**自相矛盾的决策**：`reply` 是"请回复确认"，但结构化字段却是 `readyToGenerate=true`（或 `nextAction=generate`）。

**(c) 前端和服务端都没有兜底拦截这种矛盾。**
- 服务端：`route.ts` 对 `readyToGenerate=true` 只做了"新 deck 且未选风格 → 强制风格发现"的门禁，没有"reply 在请求确认却又 readyToGenerate=true"的一致性检查，也没有"当前正在生成 → 禁止再次 generate"的状态门禁（服务端根本不知道生成是否在进行）。
- 前端：`presentation-studio.tsx:685` 的 `phase === "generating"` 排队保护只拦截**发送时刻**已处于 generating 的消息，但消息一旦进入 agent-chat 流程，等 decision 返回时（模型响应需要数秒到数十秒），`presentation-studio.tsx:747` 的 `data.readyToGenerate && data.brief → startGenerationFromBrief` 分支**不再检查 phase**，于是在原生成仍在跑（或刚结束）时直接又发起一次全量生成。

三层合起来：用户想要的是"排队一条修订"，系统实际做的是"用旧 brief 再跑一次全量生成"，且这条修订意图（+1 页对比）没有被任何持久化结构记录，最终丢失——这正是截图中的答非所问。

### 深层定性：这不是"意图识别不够智能"的问题

有一个关键观察：**截图中模型的 reply 是正确的修订提案（插入对比页、总页数 11、请求确认），说明它对用户意图的理解没有错。** 错的是同一次输出里结构化字段（readyToGenerate / nextAction）与 reply 自相矛盾，而系统只执行结构化字段。因此本问题不能靠"换更强的意图识别"解决，它由三个更底层的缺陷叠加而成：

1. **输出自洽性缺陷**：LLM 结构化输出存在偶发的字段级矛盾（reply 说"请确认"、字段说"可生成"），这是概率性的，任何模型都无法归零。而这条字段直接触发系统里最昂贵的动作（全量生成），却没有任何一致性检查。
2. **输入完整性缺陷**："生成正在进行中"这个事实根本不在请求里，模型看到的世界是"没有 deck、用户刚确认过 brief"。在这个错误的世界模型下输出"可生成"甚至不算分类错误——是输入不完整导致的必然结果。意图识别再智能，也识别不出上下文里不存在的状态。
3. **决策空间缺陷（最深层）**：生成进行中收到修订请求时，正确动作是"记下这条修订，本轮完成后再确认执行"，但 `nextAction` 枚举里不存在能表达它的值——`chat` 丢失意图、`revise-*` 因无 deckId 无法持久化为 proposal、`generate` 是灾难。**当 schema 里不存在正确选项时，模型必然选错，这是决策空间设计问题，不是识别能力问题。**

关于"门禁"的边界要区分清楚：
- **对语言做门禁**（关键字/正则猜意图）——确实永远有遗漏，本项目已明确禁止（意图归模型是既有架构不变量，维持不变）。
- **对状态做门禁**（状态机合法性检查）——"生成进行中不允许再启动生成"检查的是一个布尔事实，不解释用户的话，零遗漏零误判。它拦的不是"意图"，是"当前状态下非法的动作"，如同电梯运行中不响应开门键。

所以正确架构是三层互补：**意图归模型（不变）＋ 模型看到完整状态（让它几乎总是对）＋ 动作过状态机合法性检查（它偶发出错时，最贵路径不被点燃）**。只靠 agent 而不设状态合法性检查，等于让一个有随机性的组件独立掌握最贵动作的扳机。

### 解决方案建议

1. **【必做·补全模型输入】请求携带应用状态**：`agentChatRequestSchema` 增加 `isGenerating`（乃至更完整的 `appPhase`）字段，服务端据此注入 system 提示："一轮生成正在进行，新请求按'生成完成后的修订讨论'处理，将修改意向总结为待办提案并告知用户会在完成后确认，禁止 readyToGenerate=true / nextAction=generate。"这一条解决输入完整性缺陷，是让模型"能对"的前提。
2. **【必做·补全决策空间】让"生成中排队一条修订"成为可表达的合法输出**：两个落地方向任选其一——
   - schema 层：`nextAction` 增加 `queue-revision`（生成中提出的修订，暂存待生成完成后转正规 proposal）；
   - 编排层：生成中收到的 `revise-*` 决策不落 proposal-store（无 deckId 本来也落不了），而是由前端暂存为"待转正修订"，生成完成、artifact 就绪后自动以原始请求重放一次 agent-chat，产出可持久化的正规 proposal（前端已有 `generation-request` 排队消息雏形，`presentation-studio.tsx:685-692`，可扩展）。
   推荐编排层方案，不动 schema、与现有 proposal 生命周期兼容。可另提供"停止当前生成并按新要求重来"作为用户显式选项（中断制），但不应是默认行为。
3. **【必做·状态机合法性检查】动作执行前校验状态合法性**（注意：这是状态检查，不是意图猜测）：
   - 前端：`presentation-studio.tsx` 消费 decision 的 `readyToGenerate → startGenerationFromBrief` 与 `execute-proposal` 分支前，检查 decision 返回时刻的 `phase`——发送时刻的检查（685 行）不够，因为模型响应期间状态可能已变；
   - 服务端：`isGenerating === true` 时把 `readyToGenerate` 强制压为 false、`generate` 降级（配合方案 2 转成排队），并记日志。前后端双保险，防止竞态窗口。
4. **【建议·输出自洽性守护】decision 字段间矛盾检测**：对高成本动作增加结构自洽检查——如 `nextAction ∈ revise-*` 却 `readyToGenerate=true`、reply 含确认请求语气但 readyToGenerate=true 等结构性矛盾（前者是确定性检查可直接做；后者涉及语言判断，只记日志观测、不做拦截），矛盾时降级为 `chat` 并保留 reply。这解决输出自洽性缺陷的兜底。
5. **【建议·prompt 对齐】** 在 `presentation-brief-conversation-agent.ts` instructions 增加"generation-in-progress"策略段（与方案 1 的状态注入配套）：生成期间一律提案式回复 + 不 generate。prompt 是提升第 1 层正确率的手段，不能替代第 3 层的合法性检查。

## 两个问题的共同教训

- **校验器与生成 prompt 必须共享同一份可判定契约**：问题一是校验端的计数规则从未告知生成端；问题二是"生成中"这一状态从未告知意图分类端。都属于"链路一端知道的约束，另一端不知道"。
- **模型输出必须有确定性结构门禁兜底**：本项目已确立"意图归模型、结构校验归代码"的分工（见 `llmdoc/architecture/request-lifecycle-and-workflows.md` Invariants），问题二暴露的是 `readyToGenerate` 这条最高危路径（直接烧钱触发全量生成）反而缺少状态门禁。
- **失败/中间态的成本极不对称**：全量重生成开销巨大，任何会触发它的判定（slide 计数、readyToGenerate）都值得多一层廉价的确定性检查。
